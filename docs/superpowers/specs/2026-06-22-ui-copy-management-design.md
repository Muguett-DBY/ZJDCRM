# 全站文案管理设计

## 目标

让唯一 `admin` 账户在后台修改所有现有页面的固定界面文案，包括登录页、业务导航、业务页、后台导航、表单标签、按钮、占位提示、空状态和页面标题。修改即时生效，不需要重新部署。

## 明确边界

可修改的是界面展示层文字。例如将“线索名称”显示为“项目名称”。

不可修改的是数据库字段、接口参数、权限编码、路由、CSV 导入兼容表头和用户录入的数据值。现有 CSV 仍可使用“线索名称”导入，避免一次文案调整破坏历史流程。

## 架构

前端维护一个有稳定键名的文案注册表，每项包含 `key`、页面分组和默认中文文案。所有固定用户可见文案通过 `t(key)` 读取；未配置覆盖时返回默认值。

后台将覆盖值作为 JSON 保存在 `system_settings` 的 `ui_copy` 设置项。公共只读接口返回覆盖值；`CopyProvider` 启动时加载并向全部页面提供 `t`。管理员接口返回注册表与覆盖值，提供搜索、分组、编辑、保存和恢复默认。

## 管理界面

系统管理新增“文案管理”：

- 按登录、导航、线索、空间、导入导出、报表、个人设置和后台管理分组；
- 按键名、默认文案或当前文案搜索；
- 每行显示默认文案与当前文案，可直接编辑；
- 保存覆盖值并记录审计日志；
- 单项恢复默认，删除对应覆盖值。

## 接口与安全

- `GET /api/content/public`：返回公开界面可用的覆盖文案对象；
- `GET /api/admin/content`：返回完整注册表、覆盖文案和分组；
- `PUT /api/admin/content`：仅唯一系统管理员可保存覆盖项；
- `DELETE /api/admin/content/:key`：仅唯一系统管理员可恢复单项默认；
- 键名必须存在于注册表，值为经裁剪后的非空字符串，整体 JSON 限制 64KB；
- 所有写入需要 CSRF 并写入审计日志。

## 验证

测试覆盖公开覆盖值、管理员鉴权、非法键拒绝、保存和恢复默认。浏览器测试覆盖管理员修改“线索名称”为“项目名称”后，线索列表、表单和详情页同步显示；并覆盖文案管理页面加载。
